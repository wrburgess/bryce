# Backend Rules

**Applies to:** Backend / domain code — models, controllers, background jobs, and service objects
**Deep doc:** `docs/rules/backend-postmortems.md` (Tier 2 — deferred; read on demand when a trigger fires)

> Tier-1 Lean Core ([ADR 0004](../docs/adr/0004-two-tier-rules-layer-progressive-context.md)): always-resident invariants. Keep this file lean — push heavy, subsystem-specific case studies down to the deep doc. These are business-neutral, stack-neutral starters; **extend per host** — concrete stack-named examples live in the matching **Stack Overlay** (e.g. `ai-config-rails`), vendored alongside the baseline.

## Patterns

- **Framework & standard-library first.** Before building anything custom, prefer your framework's built-ins and the standard library, then established, well-maintained libraries. Reach for custom code only when nothing fits — and say why in the assessment/plan.
- **Thin controllers.** Keep controller actions to request/response orchestration; put domain logic in domain models, shared modules, or dedicated service/domain objects.
- **Authorize every non-public action** with the host's authorization layer — deny by default, never hardcode role checks inline.
- **Enforce invariants in the database too — declared in the schema, not only in migration SQL.** Pair model validations with DB-level constraints (`NOT NULL`, unique indexes, foreign keys, `CHECK`); a validation is not a guarantee under concurrency. Declare those constraints in the **ORM schema**, not only in hand-written migration SQL — a constraint that lives only in a migration is invisible to the schema's own diff/snapshot, so the next generated migration or table rebuild silently drops it. *(Provenance: issue #34 / PR #79; extend per host.)*
- **Anchor a freshness or completeness claim on when a job _started_, not when it _finished_.** When a job decides per-item what to include against a moving clock (e.g. "is this event final _now_?"), its finish time overstates its coverage — a run that began before a boundary may have processed early items under the old clock. Prove "this data is complete through date D" from the run's **start/claim** time versus the content boundary, and read that watermark **before** assembling the output, so a run finishing mid-read cannot forge a fresh verdict. *(Provenance: issue #34 / PR #79; extend per host.)*
- **Thread every new error type through every surface's error seam.** When you introduce a typed error, update each surface's mapping in the same change — the API error handler's status mapping, any RPC/tool layer's known-error set, and every service-layer catch that classifies errors — and add a sad-path test per surface. A seam that doesn't know the type mishandles it silently: a crash, a wrong status code, or a misleading classification. *(Host case study: `docs/rules/backend-postmortems.md`.)*
- **Confirm an external source's actual fields before coding around an assumed absence.** Before building fail-closed, degraded, or deferred behavior on the premise that a source omits a field, verify that premise against the real payload — the bundled fixture or a live sample — not the adapter's own mapping table, which shows only the fields you chose to read. "The source doesn't carry X" is a factual claim, owed the same citation discipline as any other (`rules/self-review.md`). *(Host case study: `docs/rules/backend-postmortems.md`.)*

## Anti-Patterns

- **Never introduce an implicit global query scope** that silently applies to every read, association, and new-record default — because it silently leaks into every query and is painful to bypass; use explicit, named scopes instead. *(Extend per host.)*
- **Never load a whole table into memory at once** — because iterating every row at once exhausts memory; stream large result sets in batches instead. *(Extend per host.)*
- **Never issue a query per loop iteration (an N+1)** — because it fires a query every time around the loop; preload the data or keep a maintained count instead. *(Extend per host.)*
- **Never add a heavyweight service-object framework** where a plain object would do, without a documented justification — because a plain domain object covers almost every case. *(Extend per host.)*
- **Never settle or release a durable claim/lease unconditionally** — because a worker whose lease expired and was reaped mid-flight can otherwise resurrect its own row (flip it back to a success) and overwrite the run that superseded it; gate every settle on *still owning the claim* (e.g. `WHERE status = 'running'`) and treat a no-op settle as lost ownership. *(Provenance: issue #34 / PR #79; extend per host.)*
- **Never scan or sort an append-only table on a hot path** — because an ever-growing history (delivery / run / audit rows) read in full gets slower with every insert, and a frequently polled endpoint like `/health` pays that cost forever; fetch just the latest / live / terminal rows with a suitable index and `LIMIT`. *(Provenance: issue #34 / PR #79; extend per host.)*
- **Never infer that an external source lacks a field by reading the adapter's mapping table alone** — because the table shows only what you chose to map, not what the source sends; an unmapped-but-present field then reads as "missing" and any fail-closed / degraded / deferred path built on that assumption silently drops real data. Confirm against the actual fixture or a live sample before coding around the absence. *(Provenance: issue #54 / PR #62; extend per host.)*
- **Never swap a live data file into place by moving the old one aside and then renaming the new one in** — because a crash in the gap between the two renames leaves the canonical path absent, and the next open silently creates an empty file where the real data was; rename the validated replacement *over* the destination in a single step (an atomic `rename` replaces it, so the path is never absent) and keep the prior copy elsewhere for rollback. Validate the exact, *closed* bytes you will install — never a mutable source that can still change after the check. *(Provenance: issue #67 / PR #80; extend per host.)*
- **Never re-decide a dependency's question with a proxy signal instead of the dependency's own criterion** — because the tool acts on the field *it* keys on, so a check that looks equivalent silently diverges: judging a migration "already applied" by content hash when the migrator itself orders by timestamp skips migrations a restored database still needs. Compare the same input the dependency uses. *(Provenance: issue #67 / PR #80; extend per host.)*
- **Never scope a query by materializing a set of ids and passing them to an `IN (...)` list when the set size is unbounded** — because it emits one bind parameter per id, and a large set trips the database's parameter cap (SQLite's is commonly 999), failing at runtime for exactly the big inputs that matter; it also embeds the whole id set in the SQL. Scope with a correlated `EXISTS`/join keyed on the relation instead, which stays constant-size regardless of set size and returns zero rows for an empty set with no special case. *(Provenance: issue #70 / PR #86; extend per host.)*
